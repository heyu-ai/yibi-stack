## Domain Events

過去式事件（Past tense），每個事件代表系統中一個重要的事情發生了。

1. `ReviewRoundStarted`: 一輪 review 已開始；Round 1 開始時 baseline head SHA 已記錄，Round 2 的審查面即由該 baseline 界定 — Command: lead 啟動 review round
2. `FindingReported`: reviewer voice 已提出一筆 finding，含 severity 與（或缺少）`Evidence:` 欄位 — Command: codex / agy / claude voice 產出 R1 或 R2 輸出
3. `FindingDemoted`: 一筆 finding 已被降級為非 blocking — Policy: `Evidence:` 缺漏或格式不符時觸發（結構檢查）；或 Round 2 中 severity 非 Critical 時觸發
4. `EvidenceReproduced`: Critical finding 的證據指令已執行且缺陷重現 — Command: lead 執行證據指令
5. `EvidenceRefuted`: Critical finding 的證據指令已執行但缺陷未重現 — Policy: 觸發該 finding 自 blocking 集合移除並記錄
6. `BlockingSetComputed`: 本輪的 blocking 集合已算出 — Policy: 所有 finding 完成證據判定後觸發
7. `ReviewRoundConcluded`: 一輪 review 已結束 — Policy: BlockingSetComputed 後觸發
8. `CircuitBreakerTripped`: Round 2 已結束且 blocking 集合非空，已轉交人類裁決 — Policy: Round 2 的 ReviewRoundConcluded 且 blocking 集合非空時觸發
9. `DeferralBatchIssueCreated`: 降級 Important 的批次 issue 已建立並標記 `deferred-from-review` — Policy: review 迴圈結束且存在降級 Important 時觸發
10. `MergeAuthorized`: blocking 集合為空，已授權 merge — Policy: ReviewRoundConcluded 且 blocking 集合為空時觸發

## Bounded Contexts

相關領域事件和業務規則的邊界。名稱用英文 kebab-case。

### `finding-intake`

**Responsibility**: 從各 reviewer voice 收取結構化 finding，含 severity 與證據欄位。

**Domain Events in this context**:

- `FindingReported`

**External dependencies**:

- Upstream: codex / agy CLI（read-only sandbox，context 僅 guard + prompt-r1.md + diff.patch）與 claude voice
- Downstream: `evidence-adjudication`

### `evidence-adjudication`

**Responsibility**: 判定每筆 finding 是否具備有效證據，據以決定其進入 blocking 或 deferred 集合。

**Domain Events in this context**:

- `FindingDemoted`
- `EvidenceReproduced`
- `EvidenceRefuted`
- `BlockingSetComputed`

**External dependencies**:

- Upstream: `finding-intake`
- Downstream: `round-control`（blocking 集合）、`deferral-routing`（deferred 集合）

### `round-control`

**Responsibility**: 掌管輪次狀態機、每輪的審查面界定，以及迴圈的離開條件。

**Domain Events in this context**:

- `ReviewRoundStarted`
- `ReviewRoundConcluded`
- `CircuitBreakerTripped`
- `MergeAuthorized`

**External dependencies**:

- Upstream: `evidence-adjudication`（blocking 集合）
- Downstream: 人類裁決（circuit breaker 三選項）、merge 流程

### `deferral-routing`

**Responsibility**: 將降級的 finding 送往其對應去處——Important 進批次 issue，NIT 留在既有 PR comment。

**Domain Events in this context**:

- `DeferralBatchIssueCreated`

**External dependencies**:

- Upstream: `evidence-adjudication`（deferred 集合）
- Downstream: `pr-flow:issue-triage`（承接 `deferred-from-review` 標籤的 issue）；PR comment（既有的每輪 review summary）

## Aggregate Roots

每個 Bounded Context 的一致性邊界實體。

### `Finding` (in `evidence-adjudication`)

**Entities under this aggregate**:

- `Severity` — Critical / Important / Actionable NIT 三者之一
- `Evidence` — 證據指令與其類型；得為缺漏
- `Disposition` — blocking / deferred 二者之一

**Key invariants** (business rules that must always hold):

- 一筆 finding 的 Disposition 必為 blocking 或 deferred 其一，不得同時屬於兩者，亦不得皆不屬於（降級不等於消失）
- Evidence 缺漏或格式不符的 finding，其 Disposition 必為 deferred，與 Severity 無關
- Severity 為 Actionable NIT 的 finding，其 Disposition 恆為 deferred，與 Evidence 有無無關
- Disposition 為 deferred 的 finding 必帶降級理由

### `ReviewRound` (in `round-control`)

**Entities under this aggregate**:

- `RoundNumber` — 1 或 2
- `BaselineSha` — Round 1 開始時的 head SHA
- `ReviewSurface` — 本輪實際受審的 diff 範圍
- `BlockingSet` — 本輪 Disposition 為 blocking 的 finding 集合

**Key invariants** (business rules that must always hold):

- RoundNumber 恆 ≤ 2；不存在 Round 3
- Round 1 的 ReviewSurface 為 `base..BaselineSha`（完整 diff）；Round 2 的 ReviewSurface 為 `BaselineSha..HEAD`（fix delta）。兩者為**不相交的 commit range，非子集關係**，且 Round 2 的大小不保證小於 Round 1（見 problem-frame.md W7）。終止由「不存在 Round 3」保證，不由審查面大小保證
- Round 2 的 BlockingSet 僅得含 Severity 為 Critical 的 finding
- ReviewRoundConcluded 時 BlockingSet 為空 → MergeAuthorized；非空且 RoundNumber 為 2 → CircuitBreakerTripped。兩者互斥且必居其一

### `DeferralBatch` (in `deferral-routing`)

**Entities under this aggregate**:

- `DeferredFinding` — Disposition 為 deferred 的 finding

**Key invariants** (business rules that must always hold):

- 每個 PR 至多產生一張 `deferred-from-review` issue，與降級 Important 的數量無關
- Severity 為 Actionable NIT 的 DeferredFinding 不進入 issue
- 降級 Important 數為零時不建立 issue

## Notes for Amplifier

amplifier Step 4 假設與約束的前置輸入。

- **Critical assumptions to verify in Step 4**:
  - **reviewer 有能力產生可執行的證據指令，儘管其 context 僅有 diff。** codex 與 agy 以 read-only sandbox 執行，且 guard 明文禁止其探索 repo，因此它們無從得知測試 harness 的形狀、既有測試名稱或專案的執行方式。若此假設不成立，證據閘門將把大量**真實** finding 一併降級，閘門變成「凡是外部 voice 提的都不 blocking」——與設計意圖相反。此假設對各證據類型的成立程度不同：doc 事實錯誤（路徑、file:line）與命名一致性（grep）可純由 diff 導出；test gap 的 mutation 描述可由 diff 導出；**logic / security bug 的 repro 指令最為脆弱**。
  - PR comment 會保存降級 finding —— 依賴既有的「每輪 review summary 貼為 PR comment」行為持續存在。若該行為被移除，降級 NIT 將無任何紀錄。
  - `pr-flow:issue-triage` 會實際處理 `deferred-from-review` 標籤的 issue。若無人執行 triage，Important 的降級出口即為墳場，與「不製造新累積器」的設計目標相違。
  - 2 輪足以解決多數 Critical。若實際上多數 PR 在 Round 2 後仍有 Critical，circuit breaker 將由例外路徑變成常態路徑，人類裁決的頻率上升。
  - lead 對 Critical 證據的「必跑」在實務上會被遵守。lead 為 LLM，此為 runbook 約定而非機械強制。
- **Hard constraints identified during Event Storming**:
  - reviewer 的 context 僅得為 guard + prompt-r1.md + diff.patch —— skill-hijack guard 的要求，不得為了證據閘門而放寬（放寬即重新引入 agentic 探索失效模式）。
  - codex 與 agy 以 read-only sandbox 執行，無法自行驗證其提出的證據指令是否可跑。
  - `plugins/pr-flow/skills/pr-cycle-deep/SKILL.md` 行數 ≤ 1220。
  - 三個 voice 的輸出格式已受 agentic-hijack 偵測器把關（缺 review 標題即 `[FAIL]`），新增欄位不得破壞該偵測。
- **Out of Scope items surfaced**:
  - doc / code 分流 rubric — reason: 效果已被證據閘門大部分吸收，且引入 mixed PR 判定灰色地帶與第二套待維護 rubric。
  - rules corpus 治理（`/pr-retro` 路由表出口、hot/cold tier）— reason: 不同子系統，回饋源為 `/pr-retro` 而非 review 迴圈；本 change 上線後會減少其流入量，先做本 change 可能使其變小。
  - `/pr-cycle-fast` 與 `/pr-review-cycle` — reason: 各自獨立的 SKILL.md，不在本 change 範圍。
  - 讓 reviewer 自行驗證證據可跑 — reason: 需放寬 sandbox 與 guard，違反上列硬性限制。
